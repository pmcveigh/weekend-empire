# Weekend Empire - Milestone 5 Minimal Game Simulation

Weekend Empire is a Linux-first C++20 football chairman simulator prototype.

This milestone keeps the existing SDL2 + OpenGL + Dear ImGui + SQLite foundation and adds the first meaningful simulation loop:

- in-game calendar progression
- fixture processing and result generation
- persistent club finances
- persistent board confidence
- persistent supporter mood
- practical UI controls to advance time and inspect outcomes

## What changed in this milestone

- replaced placeholder `Advance Day` behavior with real date progression saved to SQLite
- added `game_state` table for current date and selected club
- expanded `clubs` table to hold simulation values used by this milestone
- added `fixtures` table with played status and scores
- added `finance_events` table for visible and debuggable balance movement
- seeded a tiny four-club world and deterministic fixture list
- implemented day advancement that applies costs and simulates fixtures on match dates
- connected match outcomes to finances, board confidence, and supporter mood
- expanded UI with simulation sections for date, fixtures, results, and finance history
- improved runtime logging around simulation actions

## What the simulation currently includes

### Calendar progression

- game state stores a single current in-game date (`YYYY-MM-DD`)
- UI supports:
  - `Advance Day`
  - `Advance 7 Days`
  - `Refresh Data`
- each advance performs actual simulation work and persists results

### Fixtures and results

- fixtures are stored in SQLite
- each fixture has:
  - date
  - home club
  - away club
  - played flag
  - home goals
  - away goals
- when the date reaches a fixture date, unplayed fixtures for that date are simulated
- goals are generated by a simple strength-weighted Poisson model with home advantage
- simulated results are saved permanently in the database

### Finances

- selected club pays a daily running cost every simulated day
- selected club receives matchday income when involved in a played fixture
- result outcome adjusts matchday income:
  - win: better income effect
  - draw: small bonus
  - loss: reduced income
- each balance change is recorded in `finance_events` with date and reason

### Board confidence and supporter mood

- both are tracked on a 0-100 scale and clamped
- match outcomes affect both values
  - wins improve
  - losses reduce
  - home wins/losses have slightly larger supporter impact
- very weak finances apply extra pressure during daily cost application
- healthy finances can give small board stability benefit

## What is intentionally simplified

This is deliberately not a full management simulation yet.

- no player/staff database
- no tactics, transfers, injuries, or training systems
- no deep accounting model
- no full league table and no season rules engine
- fixture schedule is pre-seeded, small, and practical
- one currently selected club view

## In-game date behavior

- date is loaded from `game_state.current_date`
- pressing `Advance Day`:
  1. applies daily running cost
  2. simulates any fixtures scheduled for the new date
  3. persists the new date
- pressing `Advance 7 Days` repeats the same daily process seven times

## Fixture processing behavior

- fixtures are processed on exact date match only
- only unplayed fixtures are simulated
- simulation writes:
  - `played = 1`
  - `home_goals`
  - `away_goals`
- recent results panel displays most recent played fixtures

## Finance model behavior at this stage

- daily running cost: fixed by club (`clubs.running_cost`)
- matchday event: base income plus outcome adjustment
- finance events are written as ledger-like entries so balance movement is inspectable

## Board confidence model behavior at this stage

- baseline confidence lives in `clubs.board_confidence`
- match outcomes change confidence directly
- low post-cost cash states can reduce confidence
- strong financial position can add small positive drift

## Supporter mood model behavior at this stage

- baseline mood lives in `clubs.supporter_mood`
- match outcomes change mood
- home result swings are stronger than away swings
- poor financial pressure can slightly reduce mood

## UI now shows

- current in-game date
- selected club summary:
  - name
  - league
  - balance
  - daily running cost
  - board confidence
  - supporter mood
- upcoming fixtures
- recent results

## Dependencies (Ubuntu/Debian)

```bash
sudo apt update
sudo apt install -y build-essential cmake libsdl2-dev libgl1-mesa-dev libsqlite3-dev
```

Optional diagnostics tools:

```bash
sudo apt install -y gdb valgrind sqlite3 mesa-utils
```

## Build instructions

From repository root:

1. Configure

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug
```

2. Build

```bash
cmake --build build -j
```

## Run instructions

```bash
./build/weekend_empire
```

## Expected runtime behavior

At startup:

- SDL/OpenGL/ImGui init logs appear
- SQLite opens
- schema verification/upgrades run
- sample world and fixtures are ensured
- game state loads from SQLite

During runtime:

- window shows simulation dashboard
- advancing time changes date and club state
- fixture days produce results in recent results panel
- balance moves from daily costs and matchday events
- board confidence and supporter mood evolve over time

On shutdown:

- DB connection closes cleanly
- ImGui and SDL/OpenGL resources shut down cleanly

## Debug and troubleshooting steps

### 1) Rebuild cleanly

```bash
rm -rf build
cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build -j
```

### 2) Inspect database structure

```bash
sqlite3 weekend_empire.db ".tables"
sqlite3 weekend_empire.db "PRAGMA table_info(clubs);"
sqlite3 weekend_empire.db "PRAGMA table_info(fixtures);"
sqlite3 weekend_empire.db "PRAGMA table_info(game_state);"
sqlite3 weekend_empire.db "PRAGMA table_info(finance_events);"
```

### 3) Inspect simulation data quickly

```bash
sqlite3 weekend_empire.db "SELECT * FROM game_state;"
sqlite3 weekend_empire.db "SELECT id,name,balance,running_cost,board_confidence,supporter_mood,strength FROM clubs ORDER BY id;"
sqlite3 weekend_empire.db "SELECT id,fixture_date,home_club_id,away_club_id,played,home_goals,away_goals FROM fixtures ORDER BY fixture_date,id;"
sqlite3 weekend_empire.db "SELECT id,event_date,event_type,amount,balance_after,description FROM finance_events ORDER BY id DESC LIMIT 20;"
```

### 4) OpenGL troubleshooting

If window/context creation fails:

```bash
glxinfo | head -n 20
```

### 5) Runtime debugging tools

```bash
gdb --args ./build/weekend_empire
valgrind ./build/weekend_empire
```

## Resetting the sample database

If you want to rerun from a clean seeded world:

```bash
rm -f weekend_empire.db
./build/weekend_empire
```

This will recreate schema and seed data on next startup.
